## P2a: Keyboard Shortcuts

### Architecture

A `KeyboardShortcutProvider` wraps the app root and maintains a central registry of active shortcuts. Components register shortcuts via a `useKeyboardShortcut` hook. The registry drives a `ShortcutCheatSheet` modal opened by `?`.

### New Files

| File | Purpose |
|------|---------|
| `apps/web/src/providers/KeyboardShortcutProvider.tsx` | Context + registry + global keydown listener |
| `apps/web/src/hooks/useKeyboardShortcut.ts` | Hook for registering a shortcut |
| `apps/web/src/components/shortcuts/ShortcutCheatSheet.tsx` | Modal listing all registered shortcuts |

### Shortcut Map

| Key(s) | Action | Group | Notes |
|--------|--------|-------|-------|
| `n` | New task | Tasks | Opens new task modal |
| `e` | Edit selected task | Tasks | Requires a task to be focused/selected |
| `d` | Delete selected task | Tasks | Requires confirmation |
| `Space` | Toggle complete on selected task | Tasks | |
| `/` | Focus task search | Tasks | Already exists; registered here |
| `g` → `t` | Go to Tasks | Navigation | Sequential chord, 500ms timeout |
| `g` → `c` | Go to Calendar | Navigation | |
| `g` → `d` | Go to Dashboard | Navigation | |
| `g` → `m` | Go to Matrix (Eisenhower) | Navigation | |
| `?` | Open cheat sheet | Help | |
| `Esc` | Close modal / deselect | Global | |

### Rules

- Shortcuts are **disabled** when focus is inside `input`, `textarea`, `select`, or `[contenteditable]`
- Sequential chords use a 500ms timeout window; first key press is swallowed
- `useKeyboardShortcut` accepts `{ key, handler, description, group, disabled? }`
- Shortcuts registered by unmounted components are automatically deregistered (cleanup in `useEffect`)

### Provider Integration

`KeyboardShortcutProvider` is added to `App.tsx` wrapping the router. The cheat sheet modal is rendered once inside the provider (not per-page).

## P2b: Browser Notifications

### Data Model Change

Add `ReminderAt` (`DateTime?`, UTC) to the `Task` entity and `tasks` table.

```sql
ALTER TABLE tasks ADD COLUMN reminder_at TIMESTAMP WITH TIME ZONE NULL;
```

Included in the same EF Core migration as the labels tables (see P2c).

### API Changes

- `TaskDto` gains `reminderAt: string | null` (ISO 8601 UTC)
- `CreateTaskRequest` / `UpdateTaskRequest` accept `reminderAt?: string | null`
- `ReminderAt` is only meaningful when `DueDate` is set; backend returns a 400 if set without a due date

### Frontend — Task Form

- A datetime-local picker labelled "Remind me" appears in the task create/edit form, **only visible when a due date is already set**
- Stored and transmitted as UTC; displayed in the user's local timezone via `Intl.DateTimeFormat`

### Service Worker

**File**: `apps/web/public/sw.ts` (compiled to `sw.js`)

Responsibilities:
1. Listen for `SYNC_TASKS` messages from the main thread; store task reminder data in IndexedDB (`reminders` object store: `{ taskId, title, reminderAt }`)
2. Every 60 seconds (via `setInterval`), check IndexedDB for any entry where `reminderAt` is within the current minute and a notification has not already been fired
3. Call `self.registration.showNotification(title, { body, data: { taskId } })`
4. On `notificationclick`, post a message to open `/#/tasks?highlight={taskId}`

**Registration**: `sw.ts` is registered in `apps/web/src/main.tsx` on app load.

**Permissions**: Notification permission is requested the **first time** a user sets a reminder on a task — not on app load. If permission is denied, the reminder field is hidden with a tooltip explaining why.

### Data Flow

```
User sets reminderAt on task
  → API saves to DB
  → TanStack Query invalidates task list
  → useTasks hook posts SYNC_TASKS message to SW with updated reminders
  → SW stores in IndexedDB
  → SW setInterval fires, matches reminderAt ≈ now
  → showNotification fires
  → User clicks → navigates to task
```

## P2c: Task Labels

### Data Model

Two new tables, added in a single EF Core migration alongside `reminder_at`:

```sql
CREATE TABLE labels (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name        VARCHAR(50) NOT NULL,
  colour_hex  CHAR(7) NOT NULL,  -- e.g. '#21B8A4'
  created_at  TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
  UNIQUE (user_id, name)
);

CREATE TABLE task_labels (
  task_id   UUID NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  label_id  UUID NOT NULL REFERENCES labels(id) ON DELETE CASCADE,
  PRIMARY KEY (task_id, label_id)
);
```

### API Endpoints

**Labels resource** (`/api/v1/labels`):

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/v1/labels` | List authenticated user's labels |
| `POST` | `/api/v1/labels` | Create label `{ name, colourHex }` |
| `PUT` | `/api/v1/labels/{id}` | Update label name and/or colour |
| `DELETE` | `/api/v1/labels/{id}` | Delete label; cascades to `task_labels` |

**Task changes**:
- `TaskDto` gains `labels: LabelDto[]` (`{ id, name, colourHex }`)
- `CreateTaskRequest` / `UpdateTaskRequest` accept `labelIds: string[]`
- `GET /api/v1/tasks` supports optional `?labelId=` query param for filtering

### Backend Structure

Feature folder: `Features/Labels/` containing `LabelsController.cs`, `LabelsService.cs`, `LabelDto.cs`, `CreateLabelRequest.cs`, `UpdateLabelRequest.cs`.

Labels are always scoped to the authenticated user; service layer enforces ownership on all operations.

### Frontend

**Label management UI** (`/settings/labels` or a modal from Settings page):
- List of existing labels with coloured swatch, name, edit and delete actions
- "New label" form: text input + colour picker (12 preset colours from the design system palette)
- Inline rename and recolour

**Task form**:
- Multi-select label picker below the description field
- Renders selected labels as coloured pills/badges
- "Create new label" option inline within the picker (opens a mini form)

**Task list / cards**:
- Labels shown as small coloured pills on each task card (max 3 visible; `+N` overflow badge)
- Filter bar gains a "Label" multi-select dropdown; tasks matching **any** selected label are shown

**New files**:

| File | Purpose |
|------|---------|
| `services/labels-service.ts` | API calls for CRUD |
| `hooks/queries/useLabels.ts` | TanStack Query hook |
| `components/labels/LabelBadge.tsx` | Coloured pill component |
| `components/labels/LabelPicker.tsx` | Multi-select dropdown with inline create |
| `pages/settings/LabelsSettingsPage.tsx` | Full label management UI |

## Implementation Order

The five work items should be implemented in this sequence:

1. **WS3** — Surface rename (4 edits, ~5 minutes, lowest risk, do first)
2. **P2c** — Task labels + DB migration (longest lead time; full-stack)
3. **P2b** — Browser notifications (includes DB migration; can share migration with labels)
4. **P2a** — Keyboard shortcuts (pure frontend, no dependencies)
5. **WS4** — Launch prep (last; requires everything else to be complete and green)

Items 2–4 can be developed on separate branches in parallel if desired, merging to `develop` sequentially. The DB migration for P2b (`reminder_at`) and P2c (labels tables) should be combined into a single migration file.
